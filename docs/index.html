This project benchmarks two convolutional-neural-network architectures — DenseNet (densely-connected) and ResNet (residual) — on the task of binary gender classification from facial images. Both architectures are well-established baselines in computer vision, with distinct connectivity patterns that should manifest differently when training data is scarce.

Two datasets were used: a Kaggle gender_images training set of 100 photos (balanced 50M / 50F), and a self-curated Nawaf_data_inference set of 30 photos (15M / 15F) used to evaluate real-world generalisation. Each architecture was evaluated in two modes — training pipeline and real-time inference — producing four configurations total. The Azure ML Designer pipelines for both architectures were structurally identical except for the model component, isolating architecture choice as the sole variable.

DenseNet won both modes; the strongest configuration was DenseNet on real-time inference at 0.8667 across all three evaluation metrics — a ~7-point improvement over ResNet on the same unseen set. The findings argue that dense connectivity preserves features more effectively than residual connections on small datasets, and that real-world generalisation is a sharper test of model quality than within-distribution training accuracy.

Introduction.

Facial-image classification is one of the most-studied problems in computer vision, and gender classification is among the simplest binary variants. Yet the question of which architecture is best for small datasets — not the ImageNet-scale academic case, but the small-organisational-data case — is less settled. A 100-image Kaggle set sits in this regime.

This project sets up a tightly controlled comparison: same preprocessing, same training procedure, same evaluation protocol — only the architecture changes. DenseNet's dense connectivity and ResNet's residual connections represent two different design philosophies for combating vanishing gradients in deep networks. On a small dataset, which wins?

Research questions.

  • Which architecture — DenseNet or ResNet — generalises better on a 100-image facial dataset?
  • Does real-time inference on unseen data tell a different story than training-set evaluation?
  • Can the entire experiment be run end-to-end inside Azure ML Designer, leveraging cloud GPU compute and visual pipeline construction?

Related Work.

DenseNet — dense connectivity.

Huang et al. (2017) introduced DenseNet with the principle that every layer connects to every other layer in a feed-forward fashion. The intuition: feature reuse. By concatenating earlier feature maps into later layers, DenseNet preserves information that would otherwise be lost through deep transformation chains. This is particularly valuable when training data is scarce — features that the network is unlikely to relearn from scratch are kept available.

ResNet — residual connections.

He et al. (2016) introduced ResNet with skip connections that add (rather than concatenate) earlier outputs. The mechanism enables training of much deeper networks by giving gradients a direct path back through the architecture. ResNet became the default backbone for many computer-vision tasks and remains a strong baseline.

The small-data regime.

Most published comparisons of DenseNet vs ResNet use ImageNet (1.2M images) or larger benchmarks. Their behaviour on small datasets is less studied. Intuition from the architecture descriptions suggests DenseNet should have an edge on small data — feature reuse is more valuable when there are few examples to learn from. This project tests that intuition empirically on a 100-image facial set.

Data & Preprocessing.

Two datasets.

  • gender_images (Kaggle) — 100 photos, balanced 50M / 50F. Source: a public Kaggle dataset. Used for training and within-distribution evaluation.
  • Nawaf_data_inference (self-curated) — 30 photos, balanced 15M / 15F. Used purely for real-time inference, to test generalisation to faces the model has never seen and which weren't sampled from the same source as training.

Preprocessing held constant.

Both architectures received identical preprocessing — same Convert to Image Directory component, same 70/30 train/test split, then a further 70/30 train/val split, and the same Init Image Transformation chain applied to all three partitions. This is the foundation of the experimental design: architecture is the only variable.

Three patterns are visible in the results:

  • DenseNet wins both modes. A ~6.7-point gap in training and a ~6.7-point gap in real-time inference — consistent direction across both evaluation regimes.
  • DenseNet improves on the unseen set. 0.80 → 0.87 from training to real-time inference. This is unusual — typically generalisation is harder than within-distribution evaluation, not easier. The 30-image inference set may simply be more representative than the training-set holdout.
  • Errors are balanced across classes. Precision, recall, and accuracy all land at the same value for each configuration — male and female misclassifications happen at equal rates, not skewed.

Discussion.

Why DenseNet wins on small data.

The intuition from the architecture design holds empirically here: DenseNet's feature concatenation preserves information that residual connections might lose. On a 100-image training set, the network has too few examples to relearn lost features from scratch — every preserved feature map is valuable. ResNet's deeper architecture, designed for the abundance of data in ImageNet-scale tasks, doesn't get to express its potential.

Why real-time inference beats training-set evaluation.

DenseNet's 0.80 (training holdout) → 0.87 (real-time inference) jump is unusual. Two explanations:

  • The training-set holdout was harder. A 30-photo split from 100 mixed Kaggle photos may have included edge cases (blurry, unusual angles, low light) that the self-curated inference set doesn't have.
  • The inference set is closer to the training distribution than expected. Self-curated photos taken with consistent lighting and framing may sit more cleanly within the learned decision boundary than the messier Kaggle photos.

Either way, the headline finding — DenseNet beats ResNet by ~7 points across modes — is robust.

Why this experiment is well-designed

A common mistake in ML benchmarks is changing more than one thing per experiment. Different preprocessing, different optimiser, different architecture, all at once. The result can't isolate which change drove the metric movement.

This project deliberately holds everything constant except the architecture — same Convert to Image Directory, same 70/30 splits, same Init + Apply Image Transformation, same PyTorch training step. That's what makes the 6.7-point gap meaningful.

Limitations & Future Work.

  • Tiny dataset. 130 images total is far below modern CV-benchmark norms. A 30-image inference set is too small to claim a definitive winner — confidence intervals on these accuracies are wide. A proper benchmark would use 1000+ unseen images.
  • Single random seed. Each configuration was run once. Multiple seeds + averaged metrics would distinguish architecture effects from initialisation luck.
  • No data augmentation. Image augmentation (flips, rotations, colour jitter) typically helps small-dataset training. Future iterations should add an augmentation step to the Init Image Transformation component.
  • Binary classification only. Multi-class or fine-grained tasks (age estimation, expression recognition) may produce different architecture orderings.

Future directions.

Three obvious extensions: (i) scale the dataset to 1000+ images and re-run to confirm the architectural ordering; (ii) add a Vision Transformer (ViT) baseline to compare against modern attention-based architectures; (iii) deploy the winning DenseNet model as an Azure ML endpoint and measure inference latency under realistic API traffic.

Conclusion.

The three research questions are answered. RQ1: DenseNet wins both modes on this 100-image facial dataset, beating ResNet by ~7 points across all three evaluation metrics. RQ2: Yes — real-time inference on the unseen 30-image set tells a slightly different story than training-set evaluation; DenseNet actually improves on the inference set, a sign of strong feature transfer rather than overfitting. RQ3: Yes — the entire experiment runs end-to-end inside Azure ML Designer with no custom code, demonstrating that cloud visual-pipeline tools can match the rigour of code-first ML workflows for the right class of problem.

The practical takeaway: on small facial datasets, prefer DenseNet for feature reuse; design experiments to test generalisation, not just within-distribution accuracy; and use Azure ML Designer when the value of visual pipeline construction outweighs the flexibility of custom Python code.
